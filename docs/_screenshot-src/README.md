# Screenshot sources

This directory holds the hand-crafted sources for the panel screenshots
shown in the README and landing page. The screenshots are generated
deterministically from plain-text ANSI files, not captured from a live
terminal.

## Why not capture from tmux + fzf?

The live-capture pipeline (tmux capture-pane + freeze) has a long-standing
CJK width bug:

- `fzf` counts wide chars (Chinese ideographs) as 1 cell
- `tmux` renders them as 2 cells
- Content overflows the panel frame in `zh` screenshots
- Border segments appear discontinuous when column counts don't divide
  cleanly

A hand-written ANSI file bypasses the entire TUI pipeline. The output
is byte-for-byte reproducible: same source file ‚Üí same PNG.

## Files

| File | Role |
|------|------|
| `generate-panel.zsh` | zsh script that prints the panel as ANSI to stdout |
| `render.sh`           | one-command driver: generate ‚Üí freeze ‚Üí write `docs/screenshot-panel-*.png` |
| `panel-en.ansi`       | rendered English ANSI (regenerated by `render.sh`) |
| `panel-zh.ansi`       | rendered Chinese ANSI (regenerated by `render.sh`) |

## Regenerate

```bash
./render.sh           # both en + zh
./render.sh en        # just English
./render.sh zh        # just Chinese
```

Requires: `zsh`, `python3` (for Unicode width calculation via
`unicodedata.east_asian_width`), and `freeze`
(`brew install charmbracelet/tap/freeze`).

## Editing

To change the panel content (project names, descriptions, hotkey
labels), edit `generate-panel.zsh`:

- **Palette**: top of the file, `GREEN`/`YELLOW`/`RED`/`CYAN`/‚Ä¶
- **Layout constants**: `INNER` / `LEFT` / `RIGHT` ‚Äî be careful to keep
  `LEFT + RIGHT + 5 == INNER` so row width matches the top/bottom border
- **Projects array**: `projects=("name|status|timestamp" ‚Ä¶)`
- **Highlighted row**: `HIGHLIGHT_NAME="s1-api"`
- **Preview content**: `DESC_BODY`, `PROG_LINES`, `TODO_LINES`, `SESSIONS`
- **Localized strings**: `T_*` variables in the `en` / `zh` branches

After editing, run `./render.sh` and open the output PNGs to verify.

## Known quirks

- `display_width()` uses Python for Unicode East Asian Width lookup;
  this is ~50ms per call √ó ~100 calls per language, so generation takes
  about 1 second per language.
- Freeze's `base16` theme gives the best neutral rendering for our
  palette. Dracula/Tokyo Night themes alter the bg color in ways that
  conflict with our explicit `--background "#0b0b12"`.
- Ambiguous-width Unicode chars (`‚óè‚óê‚ñÝ‚úì‚ñ∂‚îÄ‚îÇ‚Ä¶` etc.) are rendered as 1 cell
  by `go-runewidth` (freeze's library); `display_width()` matches this
  by treating only `W`/`F` category chars as 2 cells.
